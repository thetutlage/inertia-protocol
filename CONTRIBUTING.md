# Contributing

Thanks for helping make the Inertia protocol spec clearer and more correct.

Everything you need to change lives in one file: **[`protocol.md`](./protocol.md)**.
You don't need to clone the repo, install anything, or build anything — editing
the markdown in GitHub's web editor and opening a pull request is enough. The
styled site (`index.html`) is generated by maintainers; please don't edit it.

## What this spec is (and what changes fit)

This is a **descriptive** specification of the existing Inertia v3 protocol — what
a server must send and accept so the official client (`@inertiajs/core` v3) works
unmodified. Useful contributions almost always fall into three buckets:

1. **Errata** — typos, a wrong status code, a broken cross-reference, a table cell
   or diagram that disagrees with the prose.
2. **Ambiguities** — a passage a reasonable implementer could read two ways, or a
   case the spec doesn't cover.
3. **Fidelity fixes** — the spec says one thing but the real client behaves
   differently. The client's actual behaviour wins; the spec is corrected to match.

> [!NOTE]
> **Out of scope:** proposing *new* protocol behaviour the official client doesn't
> implement. This repo documents what the client already does — new features belong
> upstream in [Inertia](https://github.com/inertiajs/inertia) first.

## Reporting an issue

Open an issue with the **Protocol concern / ambiguity** or **Erratum** template.
The most useful reports:

- cite the **section number** and quote the exact text, and
- for fidelity fixes, link the relevant client source or describe the observed
  request/response.

Not sure if it's a spec issue? Start a [discussion](../../discussions).

## Proposing a change

Edit [`protocol.md`](./protocol.md) and open a pull request. A few light authoring
conventions keep the document consistent (all are normal Markdown and render fine
on GitHub):

- **Requirement levels** — use `MUST` / `MUST NOT` / `SHOULD` / `MAY` in capitals,
  per [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119).
- **Cross-references** — write `(section 9)`, `(section 8a)`, or `(sections 10-16)`.
- **Notes / warnings** — use GitHub alerts: `> [!NOTE]` for an aside,
  `> [!CAUTION]` / `> [!WARNING]` for a hazard.
- Keep wording **precise and implementation-neutral** — describe the wire
  contract, not any one framework's API.

Changing a **diagram** (in [`diagrams/`](./diagrams)) needs local tooling; if you
can't run it, describe the change in your PR and a maintainer will render it.
Maintainers: see [CLAUDE.md](./CLAUDE.md).

### Pull request checklist

- [ ] Change is in `protocol.md` (not `index.html`).
- [ ] Cross-references and section numbers still resolve.
- [ ] Wording is precise and implementation-neutral.
- [ ] For a fidelity fix: the client's real behaviour is cited or described.
